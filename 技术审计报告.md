Financial Asset QA System 是一个面向金融领域的智能问答系统，通过 LLM + Tool Use 架构实现：

### 2.2 AI Agent 架构评估

**工具清单** (6 个工具):

         ↓

         ↓

         ↓

### 2.3 RAG 系统架构

**实现方案**: 两阶段检索 + 混合检索

**组件**:
1. **RAGPipeline** (pipeline.py, 127 行)
   - Bi-Encoder: bge-base-zh-v1.5 (向量检索)
   - Cross-Encoder: bge-reranker-base (精排)
   - Top-K=10 → Top-N=3, score > 0.7

2. **HybridRAGPipeline** (hybrid_pipeline.py)
   - BM25 关键词检索
   - 向量语义检索
   - 混合权重: 0.3 (BM25) + 0.7 (Vector)

3. **ConfidenceScorer** (confidence.py)
   - 计算检索置信度
   - 基于分数分布和文档一致性

**评分**: 8.5/10

**优点**:
- ✅ 两阶段检索设计合理，平衡召回和精度
- ✅ 混合检索支持关键词和语义双重匹配
- ✅ 置信度评分机制，可用于判断是否需要补充搜索
- ✅ ChromaDB 嵌入式部署，简化运维

**不足**:
- ⚠️ 知识库规模小（约 8000 字），实用性有限
- ⚠️ 缺少文档分块策略说明
- ⚠️ 未实现增量更新机制
- ⚠️ 缺少时间感知（无法区分过时信息）

### 3.2 Tool Use 设计

**工具定义**: JSON Schema 格式，符合 Anthropic 规范

**工具执行流程**:
1. LLM 决策调用哪个工具
2. AgentCore 解析工具参数
3. 执行工具（异步）
4. 返回结果给 LLM
5. LLM 生成最终回答

**工具调用示例**:
```json
{
  "name": "get_price",
  "input": {"symbol": "BABA"}
}
→ MarketDataService.get_price("BABA")
→ {price: 89.52, currency: "USD", ...}
```

**评分**: 9/10

**优点**:
- ✅ 工具定义清晰，描述详细
- ✅ 参数验证完善（Pydantic）
- ✅ 错误处理机制
- ✅ 工具执行日志记录

**不足**:
- ⚠️ 未实现工具并行执行（文档提到可优化）
- ⚠️ 缺少工具调用超时控制

### 3.3 Streaming 机制

**实现方式**: SSE (Server-Sent Events)

**事件类型**:
- `model_selected` - 模型选择通知
- `tool_start` - 工具开始执行
- `tool_data` - 工具返回数据
- `chunk` - 文本片段（逐 token）
- `done` - 完成
- `error` - 错误

**前端处理**:
```javascript
fetch('/api/chat', {method: 'POST'})
  .then(response => response.body.getReader())
  .then(reader => {
    // 逐行解析 SSE 事件
    // 实时更新 UI
  })
```

**评分**: 9.5/10

**亮点**:
- ✅ 完整的 SSE 实现，用户体验优秀
- ✅ 事件类型丰富，前端可精细控制 UI
- ✅ 工具执行状态实时反馈
- ✅ 错误处理完善

### 3.4 QueryEnricher 设计

**实现**: 零成本规则引擎（无 LLM 调用）

**工作原理**:
```python
关键词检测 → 注入提示 → 增强查询

"阿里巴巴股价"
→ [Hint: 涉及行情数据，优先使用 get_price 工具]
→ 提升工具调用准确率
```

**关键词库**:
- 行情关键词: 价格、股价、涨跌、市值...
- 知识关键词: 什么是、如何、定义...
- 新闻关键词: 为什么、原因、新闻...

**评分**: 8/10

**优点**:
- ✅ 零成本优化，无额外 API 调用
- ✅ 提升工具选择准确率
- ✅ 实现简单，易于维护

**改进建议**:

## 7. 风险与问题识别

### 7.1 架构风险

| 风险项 | 严重程度 | 影响 | 建议 |
|--------|---------|------|------|
| 单点故障 | 🔴 高 | Redis/ChromaDB 故障导致服务不可用 | 添加健康检查和自动重启 |
| 数据源依赖 | 🟡 中 | yfinance 免费 API 不稳定 | 接入商业数据源 |
| 无认证机制 | 🔴 高 | API 可被滥用，成本失控 | 添加 API Key 认证 |
| 无限流保护 | 🟡 中 | 恶意请求导致资源耗尽 | 实现 Rate Limiting |
| 缺少监控 | 🟡 中 | 故障无法及时发现 | 集成 Prometheus/Grafana |

### 7.2 性能瓶颈

**识别的瓶颈**:

1. **工具串行执行**
   - 问题: 多工具调用时延迟累加
   - 影响: 用户等待时间长
   - 解决: 实现并行执行（asyncio.gather）

2. **模型加载时间**
   - 问题: 首次启动加载 bge 模型需 30-60 秒
   - 影响: 冷启动慢
   - 解决: 模型预热或使用模型服务

3. **前端长列表渲染**
   - 问题: 对话超过 50 条时卡顿
   - 影响: 用户体验下降
   - 解决: 虚拟滚动

4. **Redis 单点**
   - 问题: Redis 故障导致所有请求变慢
   - 影响: 性能下降 10x
   - 解决: Redis 集群或降级策略

### 7.3 安全问题

**已识别的安全风险**:

1. **无身份认证** 🔴
   - 任何人可访问 API
   - 建议: JWT 或 API Key

2. **无输入验证** 🟡
   - 可能存在注入攻击
   - 建议: 严格的输入校验

3. **API Key 明文存储** 🟡
   - .env 文件可能泄露
   - 建议: 使用密钥管理服务

4. **无 HTTPS** 🟡
   - 本地开发未配置 SSL
   - 建议: 生产环境强制 HTTPS

5. **CORS 过于宽松** 🟡
   - 允许所有来源
   - 建议: 限制允许的域名

6. **无请求签名** 🟡
   - 无法防止重放攻击
   - 建议: 添加时间戳和签名

### 7.4 数据可靠性问题

**数据质量风险**:

1. **yfinance 数据延迟**
   - 延迟约 15 分钟
   - 影响实时性要求高的场景

2. **无数据校验**
   - 异常价格（如负数）未过滤
   - 可能误导用户

3. **知识库过时**
   - 静态知识库无更新机制
   - 信息可能过时

4. **缓存一致性**
   - 缓存过期策略可能导致数据不一致
   - 建议: 添加版本号或主动失效

### 7.5 可扩展性问题

**扩展瓶颈**:

1. **ChromaDB 嵌入式部署**
   - 单机存储限制
   - 无法水平扩展
   - 建议: 迁移到 pgvector 或 Qdrant

2. **无分布式架构**
   - 单实例部署
   - 无法负载均衡
   - 建议: 容器化 + K8s

3. **会话状态内存存储**
   - 重启丢失
   - 无法跨实例共享
   - 建议: Redis 存储会话

4. **模型本地加载**
   - 每个实例加载模型占用内存
   - 建议: 模型服务化
